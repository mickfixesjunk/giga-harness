# giga-bootstrap-project

> **For most single-host projects, just run `giga setup` from your project directory.** It launches a Claude Code session with a baked-in bootstrap prompt that asks five questions and scaffolds everything end-to-end (config at `~/.giga/configs/<name>/`, per-agent CLAUDE.md, registry entry in `~/.giga/swarms.toml`). This skill remains the reference for the multi-host / multi-developer layouts described below, which `giga setup` doesn't yet handle.

How to lay out your giga config so it scales from "one box" to "multiple developers, multiple hosts, multiple ecosystems" without painful migrations.

## Decision: which layout fits?

Ask the user a single question to pick:

| Their answer | Recommended layout |
|---|---|
| "I'll run this on multiple machines / share with teammates" | **A. Dedicated configs repo (recommended)** |
| "One project, one machine, just me" | **B. Subdirectory in the main project** |
| "I'm experimenting — don't care about versioning yet" | **C. Local files, no git** |

All three follow the same internal structure (`giga-harness.toml` + `agents/` + `setup-*.sh`). The difference is where the directory lives.

## A. Dedicated configs repo (recommended)

This is what the canonical reference setup uses (`giga-harness-configs`). Best when configs need to travel across machines or be shared with other humans.

```
~/projects/<org>/giga-harness-configs/        # one git repo per org/user
  .gitignore                                  # ignore generated files
  README.md
  my-project/                               # one subdir per project/ecosystem
    README.md                                 # docs the agent layout + history
    giga-harness.toml                         # canonical config (commit this)
    setup-<host>.sh                           # one per host (commit these)
    agents/                                   # canonical templates (commit these)
      my-project.md
      design.md
      ...
    giga-harness.<host>.toml                  # generated, GITIGNORED
    agents.<host>/                            # generated, GITIGNORED
      ...
  <next-project>/
    ...
```

`.gitignore` boilerplate:
```
# Per-host localized configs and templates generated by setup-*.sh.
# Machine-specific paths ($HOME, Windows username, etc.) — don't commit.
**/giga-harness.*.toml
**/agents.*/
```

**Per-host setup script (`setup-<host>.sh`):**
- Clones any source repos this ecosystem needs (the agents' workdirs)
- Installs giga (Linux side via install.sh, Windows side via install.ps1)
- Auto-detects machine-specific values (Windows username, $HOME)
- Localizes `giga-harness.toml` → `giga-harness.<host>.toml`, substituting canonical paths (e.g. `/home/alice/` → real `$HOME`, `C:\Users\Alice\` → `C:\Users\<actual-user>\...`)
- Same substitution applied to each `agents/<slug>.md` → `agents.<host>/<slug>.md`
- Drops a `giga-harness.toml` symlink (WSL) or copy (Windows) into each agent's workdir
- Merges `bypassPermissions` into the Windows-side claude settings
- Runs `giga init` to scaffold inbox files + render CLAUDE.md files

**Why a separate repo (not a subdir of one of your source repos):**
- A configs repo holds N projects' configs side-by-side
- The agents in one project might span M source repos (engine + lib + frontend); none of those is the natural "home"
- Easy to share with collaborators without granting access to your source code

**Bring-up on a new machine:**
```sh
gh repo clone <org>/giga-harness-configs ~/projects/<org>/giga-harness-configs
~/projects/<org>/giga-harness-configs/<project>/setup-<host>.sh
~/.local/bin/giga launch ~/projects/<org>/giga-harness-configs/<project>/giga-harness.<host>.toml
```

## B. Subdirectory in the main project

Best when you have ONE project and don't need multi-host portability. Drop a `.giga/` directory at the root of the main source repo:

```
<your-project>/                # e.g. ~/projects/foo
  src/
  ...
  .giga/
    giga-harness.toml
    agents/
      <slug>.md
```

Trade-offs:
- Faster to find: the agents config sits next to the code they work on
- Versioned alongside the code
- BUT: if your ecosystem has multiple source repos, only one of them can host `.giga/`
- BUT: no per-host localization story — you commit machine-specific paths or write a `.giga/setup.sh` yourself

For a solo project on one machine: fine. For more than one host or more than one developer: outgrow it quickly.

## C. Local files, no git

Loose files under `~/giga/`:

```
~/giga/
  <project>/
    giga-harness.toml
    agents/
      <slug>.md
```

Run from cwd: `cd ~/giga/<project> && giga init && giga launch`. No portability, no version history. Use only for experiments. Promote to A or B when the experiment outlasts a few sessions.

## Scaffolding a fresh project

Once layout is chosen, scaffold the new project dir:

1. **Create the project dir** at the chosen location.
2. **Write `giga-harness.toml`** — start from `<giga-harness-source>/examples/minimal/giga-harness.toml`. Fill in:
   - `[project]` — name and description
   - `[paths]` — inbox dirs for wsl and/or windows
   - At least one `[[agents]]` entry
   - At least one `[[channels]]` entry per pair of agents that talk
   - `[bench_protocol]` if any agent does heavy work
3. **Create `agents/`** with one `<slug>.md` per agent. Use the `giga-add-agent` skill to scaffold each — it ensures all three artifacts (TOML, template, channels) stay in sync.
4. **Write `setup-<host>.sh`** if going with layout A. Copy from an existing example and adjust the localization substitutions for your project's path conventions.
5. **Run `giga validate <config>`** before init. Catches typos in channel participants, missing inbox dirs, multiple bench schedulers, etc.
6. **Run `giga init <config>` then `giga launch <config>`**. Init is idempotent; safe to re-run.

## Naming conventions worth following

- **Project dir name**: kebab-case, matches `[project].name` in TOML.
- **Channel filenames**: `<participant-a>-<participant-b>.md`, alphabetical or activity-flow order (look at existing entries in the repo for the local convention).
- **Agent slugs**: kebab-case, short, role-focused (`engine`, `web`, `deploy`), not platform-focused (avoid `linux-tester` if a name like `tests` says what it does without coupling to the OS).
- **Per-host setup script**: `setup-<hostname>.sh` if you have a few well-known hosts; `setup.sh` if there's just one canonical bring-up. The script's substitution table is the actual per-host config — the filename is just a label.

## What NOT to do

- **Don't store configs inside one of the agents' workdirs.** That workdir's `CLAUDE.md` will get clobbered on `giga init`; mixing the harness config with the agent's own working files makes both harder to reason about.
- **Don't commit localized files.** `agents.<host>/`, `giga-harness.<host>.toml`, the per-workdir symlinked `giga-harness.toml` — all gitignore.
- **Don't share secrets via channel files.** Channels are plain markdown on disk. Use the host's secret manager (AWS Secrets Manager, 1Password, env vars) and have agents look them up at runtime.
- **Don't skip the per-host setup script and try to write machine paths into the canonical config.** Even if you're only on one machine today, the canonical/localized split makes adding the second machine a 5-minute job instead of a half-day migration.
