# giga-add-agent

Scaffold a new agent into an existing giga-harness ecosystem. This skill encodes the steps to keep `giga-harness.toml`, `agents/<slug>.md`, and `[[channels]]` consistent — getting them out of sync is the most common way to wedge a project.

> When the new agent shares a codebase with existing ones, pass `--code-root <path>` so the agent gets a clean workdir but edits in the shared code location. The bilateral and broadcast channels are still set up the same way.

## When to use

User wants to add an agent to a giga ecosystem. Phrasings:
- "Add a new agent for the website/backend/whatever"
- "Scaffold a [role] agent in your-swarm"
- "I have a new agent that does X, integrate it into giga"

## Pre-flight: confirm you're in a giga project

Required files in cwd (or in a subdir the user names):
- `giga-harness.toml` — canonical config
- `agents/` — canonical templates dir

If not, ask which project dir to operate on, or refuse and explain.

**Never edit `giga-harness.*.toml` (localized) or `agents.*/` (generated).** Those are per-host artifacts produced by `setup-*.sh`. Edits there get clobbered on next setup. The canonical files you touch are `giga-harness.toml` and `agents/*.md`.

## Gather requirements (one AskUserQuestion batch)

You need these, ideally in a single AskUserQuestion call:

1. **Slug** — kebab-case, short. Shows up in channel filenames (`<slug>-your-swarm.md`) and `giga post --as <slug>`. Examples: `web`, `deploy`, `frontend`, `infra`.
2. **One-line role** — for the `role = "..."` field in TOML. Concrete; what does this agent OWN?
3. **Workdir** — absolute path on the agent's target OS. WSL agents: `/home/<user>/projects/...` or wherever the repo lives. Windows agents: `C:\Users\<user>\...`. If a separate Windows-native agent, this needs to be a Windows path.
4. **Platform** — `wsl` or `windows`.
5. **Peers** — names of existing agents this one should have bilateral channels with. Read the existing `[[agents]]` table from `giga-harness.toml` and present them as options. The "engine" / "scheduler" agent (the one with `bench_scheduler = true`) is almost always needed.
6. **Bench scheduler participation** — does this agent's heavy work need to clear with the bench scheduler? Most ops/web agents don't; perf-sensitive agents do.

Use a single AskUserQuestion with the questions batched. For peers, use multiSelect.

## Generate

### 1. `[[agents]]` entry in `giga-harness.toml`

Append after the last existing `[[agents]]` block, **before** any `[[channels]]` blocks. Format:

```toml
[[agents]]
name = "<slug>"
workdir = "<workdir>"
role = "<one-line role>"
platform = "<wsl|windows>"
claudemd_template = "agents/<slug>.md"
```

For Windows workdirs, escape backslashes (TOML basic-string convention): `"C:\\Users\\Alice\\..."`.

If the canonical `giga-harness.toml` uses placeholder values for the canonical author's machine (e.g. `Alice` for the Windows user, `/home/alice/` for the WSL home), match that convention. The localizer in `setup-<host>.sh` substitutes these at install time. **Don't bake in the current user's actual paths in canonical files.**

### 2. Canonical template at `agents/<slug>.md`

Use this skeleton. Fill the `{...}` placeholders. Don't include the giga-generated `<!-- Generated by giga-harness... -->` header — that gets added at install time when the template is rendered into the agent's workdir.

```markdown
# <slug> agent

You are the **<short role description>** for the <project-name> project.

## Session Start (do this first, every session)

**Step 0 (always):** If `./HANDOVER.md` exists in your workdir, read it before anything else. It carries cross-session / cross-machine state — recent decisions, in-flight work, pickup instructions — that your conversation history may not include. If a previous session left pickup instructions, follow them.

In order:

1. **Post intro** on each of your channels via `giga post <channel> --as <slug> --subject "online" --body "<slug> session resumed — <one-line status>"`. Informational.
2. **Arm the Monitor below.** Use the exact `Monitor(persistent: true, command: ...)` invocation — don't paraphrase. One watcher; auto-discovery handles the channel list.
3. **Standby for messages.** Don't initiate new work without being asked. If you were mid-task per HANDOVER.md or your prior session, finish that first.

## Your responsibilities

- {what this agent OWNS — concrete deliverables, files, infra}
- {who they answer to / who answers to them}
- {any standing rules or invariants}

## Channels you watch

```
Monitor(persistent: true, command: "giga watch --as <slug>")
```

One watcher auto-discovers every channel where you're a participant (per `giga-harness.toml`). New channels added later — e.g. when a new peer agent comes online — get picked up automatically on the next config reread (~15s). Notifications are formatted `inbox <channel>: [sender] ...`.

## Convention

Every channel message ends with either:

* `WAITING ON: <agent> (<what's needed>)` — if a reply is expected.
* `Informational, no response required.` — otherwise.

Ambiguous closings stall the pipeline. Use the tag.
```

If the agent participates in bench coordination, add a `## Bench coordination` section pointing at the scheduler agent. Copy the language from an existing agent's template so terminology stays consistent across the ecosystem.

### 3. `[[channels]]` entries

One per peer. Append at the bottom of the channels section in `giga-harness.toml`. File naming convention: alphabetical or `<new>-<existing>.md` (look at existing entries — e.g., `design-your-swarm.md` is design-first because design is alphabetically earlier).

```toml
[[channels]]
file = "<slug>-<peer>.md"
side = "wsl"  # or "windows" if both participants are Windows-platform
participants = ["<slug>", "<peer>"]
purpose = "<one-line: what this channel is for>"
```

Choose `side` based on where the channel file should live:
- `wsl` for WSL ↔ WSL bilateral or anything that primarily flows through the WSL side
- `windows` only if BOTH participants are Windows-platform agents (rare)

### 4. Add to the broadcast channel (if one exists)

Some projects have a broadcast/announcement channel (typically `_broadcast.md` — the underscore prefix is the convention) where all agents are listed as participants. Check the existing config for any channel whose filename starts with `_` or whose `participants` list includes most/all agents. If found, append `<slug>` to that channel's `participants` array.

```toml
# before
participants = ["alice", "bob", "carol", "dave"]

# after
participants = ["alice", "bob", "carol", "dave", "<slug>"]
```

This way the new agent's auto-discovery watcher picks up broadcast announcements the first time it starts, with no extra wiring. Skip this step if the project has no broadcast channel.

### 5. Update README (if one exists)

If the project has a README.md mentioning the agent count or listing agents, update both. Examples:
- "8-agent ecosystem" → "9-agent ecosystem"
- Table of agents in the README — add a row

## Apply

Tell the user:

```
Done. To apply on this host:

  <project-dir>/setup-neo.sh          # or your project's setup script — re-localizes config + templates
  ~/.local/bin/giga launch --only <slug> <project-dir>/giga-harness.<host>.toml

New tab joins the existing wt window / tmux session for <slug>; the other agents keep running undisturbed.
```

`--only` (added in 0.1.9) is what makes this non-disruptive. A bare `giga launch` re-spawns every tab (and nukes the prior tmux session); use it only on a cold start.

If the user has torn the original launch window apart (one window per agent, arranged on screen) ask whether to add `--new-window` so the new agent gets its own standalone wt window instead of docking into a residual `giga-<project>` window.

If the project has no `setup-*.sh` (i.e. simpler single-host setup):

```
  giga init <project-dir>/giga-harness.toml
  giga launch --only <slug> <project-dir>/giga-harness.toml
```

## Verify before reporting done

Run two sanity checks before telling the user it's ready:

1. `giga validate <config>` — confirms the TOML parses, every channel participant is a known agent, sides have inbox dirs configured.
2. `grep -c '^\[\[agents\]\]' <config>` — agent count matches what you intended.

If validate fails, the error tells you exactly what's broken. Fix before reporting.

## Common mistakes to avoid

- **Editing the localized config** (e.g. `giga-harness.neo.toml`) instead of canonical `giga-harness.toml`. Localized configs are regenerated.
- **Editing `agents.<host>/<slug>.md`** instead of `agents/<slug>.md`. Same reason.
- **Forgetting one of the three artifacts.** Agent without a channel is a tab nobody talks to. Channel without an agent participant fails validate. Template without a TOML entry never gets rendered.
- **Hardcoding the current user's paths** in canonical files. Use the canonical author's placeholders (`/home/alice/`, `C:\Users\Alice\`) so the localizer can substitute them per-host.
- **Skipping the README update.** People skim the README to understand the ecosystem; a stale agent count or table is misleading.
